# Access With Token

회사 프로젝트에 반영하기 전에 쿠키 기반 토큰 인증으로 페이지 접근을 제어하는 흐름이 정상적으로 동작하는지 검증하기 위해 작성한 예제입니다.

## 전체 흐름

```
보호된 페이지 접근 시도 (토큰 없음)
  → 인증 폼으로 redirect (redirect 파라미터에 원래 경로 저장)
    → 폼 제출 → /callback으로 이동
      → 미들웨어가 가로채서 쿠키 세팅 + 보호된 페이지로 redirect
        → 서버에서 쿠키 확인 → 페이지 렌더링
          → CookieCleaner가 쿠키 즉시 삭제 (일회성)
```

## 파일 구조

| 파일 | 설명 |
| --- | --- |
| `page.tsx` | 메인 페이지 — "인증 페이지"와 "보호된 페이지" 링크 제공 |
| `form/page.tsx` | 인증 폼 — 전화번호 + 인증번호 입력 후 `/callback`으로 이동 |
| `protected/page.tsx` | 보호된 페이지 — 서버에서 쿠키 확인, 없으면 인증 폼으로 리다이렉트 |
| `components/CookieCleaner.tsx` | 클라이언트 컴포넌트 — 마운트 시 쿠키 즉시 삭제 (일회성 접근) |
| `constants.ts` | 쿠키 키 상수 (`authToken`) |
| `middleware.ts` (src/) | `/callback` 요청을 가로채서 토큰 쿠키 세팅 + 리다이렉트 처리 |

## 핵심 테스트 포인트

- **미들웨어를 인증 콜백으로 활용**: 별도 API Route 없이 미들웨어만으로 쿠키 세팅 + 리다이렉트 처리
- **서버 사이드 쿠키 체크 + redirect**: `cookies()` + `redirect()`로 미인증 사용자 차단
- **일회성 토큰 접근**: `CookieCleaner`가 페이지 진입 시 쿠키를 삭제하여 새로고침하면 재인증 필요
- **캐싱 완전 비활성화**: `force-dynamic`, `revalidate = 0`, `force-no-store` 설정으로 항상 서버에서 최신 쿠키 상태 확인
- **redirect 쿼리 파라미터**: 인증 후 원래 가려던 페이지로 돌아가는 흐름

## 참고 사항

실제 회사 프로젝트에 반영할 때는 Next.js의 Router Cache가 남아 있어서 쿠키를 삭제해도 캐시된 페이지가 그대로 보이는 문제가 있었습니다. 이를 해결하기 위해 별도의 캐시 무효화 처리가 추가로 필요했습니다.
